# SIEM Enterprise: Мониторинг комплаенса и банковской аналитики (Kaggle Fraud Dataset)

Интерактивный аналитический комплекс на фреймворке Streamlit, разработанный для специалистов служб информационной безопасности (ИБ) и департаментов комплаенс-контроля. Система осуществляет потоковый дата-майнинг транзакционных логов, производит многокритериальную фильтрацию аномалий и автоматизирует выгрузку регламентированной отчетности.

## Архитектурные возможности и бизнес-логика

- **Глобальный комплаенс-контроль:** Левое исполняющее меню (сайдбар) выступает в роли глобального конфигуратора базы данных. Изменение порогов риска или каналов эквайринга мгновенно пересчитывает KPI-метрики, финансовые результаты и графики платформы на всех вкладках.
- **Выплывающая панель дата-майнинга:** Все инструменты локального поиска упакованы в компактный выплывающий модуль (`st.expander`), что разгружает интерфейс и оставляет экран свободным для макро-аналитики.
- **Вертикальные интерактивные диапазоны:** Слайдеры числовых метрик (суммы, аппаратные риски, попытки авторизации) выстроены вертикальным столбиком. Это решает проблему тесноты интерфейса, расширяет область ползунков для работы с длинными числами и исключает наложение текста.
- **Умная автоматизация отбора:** Реализована система потокового проставления флагов. Оператор может в один клик автоматически отметить галочками весь отфильтрованный срез, а затем точечно скорректировать выборку вручную перед выгрузкой.
- **Функциональная печать и экспорт:** Система генерирует изолированные микро-выборки и выгружает их в CSV. Графики Plotly снабжены нативным браузерным движком печати (иконка фотоаппарата в углу диаграмм) для мгновенного сохранения PNG без нагрузки на сервер.
- **Генератор регламентированной отчетности:** Автоматическое составление печатных текстовых форм на основе активного среза данных:
  - Внутренний конфиденциальный рапорт для Службы безопасности банка.
  - Официальный доклад для ЦБ РФ в соответствии с Положением № 683-П (требования ФинЦЕРТ).
  - Специальное сообщение для Росфинмониторинга по форме Федерального закона № 115-ФЗ.

## Спецификация структуры данных (Kaggle Dataset)

Приложение обрабатывает следующие ключевые параметры транзакционных сессий:


| Название столбца | Отображение в UI | Описание бизнес-сущности |
| :--- | :--- | :--- |
| `transaction_amount` / `сумма` | **Сумма операции** | Финансовый объем транзакции в долларах США ($). |
| `device_risk_score` / `риск_устройства` | **Уровень аппаратного риска** | Индекс компрометации гаджета (0-1). Выше 0.65 — эмуляторы и фрод. |
| `login_attempts` / `попытки_входа` | **Попытки авторизации** | Количество попыток ввода учетных данных в рамках сессии. |
| `payment_method` / `способ_оплаты` | **Канал эквайринга** | Платежный инструмент (Credit Card, Crypto, СБП и др.). |
| `authentication_method` / `подтверждение` | **Защита сессии** | Протокол верификации (SMS OTP, Биометрия, Без подтверждения). |
| `transaction_type` / `тип_транзакции` | **Тип операции** | Категория перевода (Онлайн-покупка, Перевод физлицу). |
| `is_fraud` / `статус` | **Вердикт комплаенса** | Итоговое решение системы (Легитимная / Мошенническая). |

*Примечание:* Если физический файл `banking_transactions.csv` отсутствует в корневой директории, система автоматически запускает генератор математически сбалансированных синтетических логов (бета- и экспоненциальные распределения) объемом в 5000 сессий для демонстрации работы SIEM.

## Инструкция по развертыванию и запуску

Приложению для стабильной работы требуются библиотеки версии `Streamlit >= 1.35.0` (для корректной работы обновленного синтаксиса `st.rerun` и интерактивного табличного редактора `st.data_editor`).

### Быстрый запуск через Makefile:
1. **Установка окружения и зависимостей:**
   ```bash
   make install
   ```
2. **Запуск локального сервера аналитики:**
   ```bash
   make run
   ```

### Ручной запуск в терминале:
1. Создайте виртуальное окружение и активируйте его:
   ```bash
   python -m venv venv
   source venv/bin/activate  # Для Linux/macOS
   .\venv\Scripts\activate   # Для Windows
   ```
2. Установите необходимые пакеты:
   ```bash
   pip install -r requirements.txt
   ```
3. Запустите комплаенс-монитор:
   ```bash
   streamlit run app.py
   ```
4. Откройте сгенерированный локальный адрес в браузере (по умолчанию: `http://localhost:8501`).

## Инструкция для синхронной работы аналитика

Для получения строго валидных данных при расследовании кибератак соблюдайте **правило сквозной синхронизации**:
1. Выставите базовые макро-параметры (например, канал `Криптовалюта` и риск `> 0.50`) в левом сайдбаре, чтобы перестроить общие финансовые KPI платформы.
2. Раскройте центральную выплывающую панель и с помощью вертикальных слайдеров сузьте выборку до точного диапазона цен или попыток входа.
3. Активируйте чекбокс автоматического выбора строк, проверьте сформированную микро-выборку и нажмите кнопку печати отчета для выгрузки вещественных доказательств.

### Актуальность и проблематика
В условиях стремительного роста объемов цифровых платежей и усложнения векторов кибератак (социальная инженерия, использование ботнетов, эмуляторов устройств), традиционные транзакционные фильтры перестают быть эффективными. Аналитикам ИБ и комплаенс-контроля требуется инструмент, способный проводить потоковый дата-майнинг, оперативно локализовать аномалии и мгновенно формировать доказательную базу инцидентов для регуляторов.

### Цель и назначение разработки
**SIEM Enterprise Analytics** — это программный комплекс класса SIEM (Security Information and Event Management), реализованный в виде интерактивного антифрод-монитора. Цель проекта — автоматизация процессов верификации транзакционных логов, оценка уязвимости каналов эквайринга, расчет предотвращенного ущерба банка и генерация регламентированной юридической отчетности. 

### Научная и техническая новизна
- **Двухуровневая архитектура фильтрации:** Разделение контуров управления на макро-уровень (глобальные фильтры сайдбара, перестраивающие KPI-метрики всей системы) и микро-уровень (выплывающий интерактивный конструктор логов для точечной сборки локальных дата-срезов).
- **Эргономичный UI/UX рефакторинг:** Чистый, строгий интерфейс, оптимизированный под работу аналитика. Использование вертикально ориентированных слайдеров решает проблему тесноты экрана и искажения числовых диапазонов при масштабировании окон.
- **Отказоустойчивое ядро:** Реализована сквозная проверка существования переменных памяти и безопасное вычисление финансовых KPI (агрегация за один проход по датафрейму, динамическое сжатие миллионов в компактный формат `$1.24M`).
- **Автоматизация комплаенс-экспорта:** Интегрирован механизм умного массового проставления чекбоксов, позволяющий автоматически формировать выгрузки отчетов в один клик.

### Практическая значимость и результаты
Комплекс позволяет снизить время расследования ИБ-инцидента (Time to Detect / Time to Respond) в 3–4 раза. Интегрированный аналитический генератор рапортов автоматически формирует документы по стандартам Положения ЦБ РФ № 683-П (требования ФинЦЕРТ) и Федерального закона № 115-ФЗ (Росфинмониторинг).

## Самое интересное 

## РАБОЧИЕ СЦЕНАРИИ ИБ-АНАЛИТИКА (10 ПРАКТИЧЕСКИХ КЕЙСОВ)

Левая панель управления (сайдбар) и выплывающий конструктор кастомизации позволяют моделировать, локализовать и расследовать 10 критических сценариев нагрузок и киберугроз:

### Сценарий 1: Распределенный брутфорс авторизации на криптовалютном шлюзе
- **Вектор атаки:** Хакерская сеть ботнетов осуществляет автоматизированный перебор паролей для вывода фиатных средств через шлюз `Crypto`.
- **Маркеры аномалии:** Пакетный вброс транзакций с аппаратным риском `0.92 – 1.00`, количеством попыток входа `5 – 8 ед.` и методом подтверждения `None`.
- **Действия аналитика:** Фиксация красного регистра предупреждения `ОБНАРУЖЕНА АТАКA`, массовое выделение скомпрометированных строк и скачивание CSV для немедленной блокировки IP-адресов на сетевом шлюзе.

### Сценарий 2: Поведенческие искажения при массовом использовании VPN-сервисов
- **Суть инцидента:** Легитимные клиенты банка массово включают коммерческие VPN/прокси для обхода внешних геоблокировок.
- **Маркеры аномалии:** Смещение индекса аппаратного риска устройств в среднюю «серую» зону (`0.36 – 0.65`). Объемы операций и частота входов остаются в пределах нормы.
- **Действия аналитика:** Настройка слайдера бизнес-моделирования для поиска баланса: повышение порога жесткой блокировки с целью минимизации ложных тревог (False Positive) и предотвращения необоснованной заморозки счетов лояльных пользователей.

### Сценарий 3: Атака типа ATO (Account Takeover) через перехват SMS-кодов
- **Вектор атаки:** Злоумышленники используют уязвимости протокола SS7 или дубликаты SIM-карт (SIM Swapping) для перехвата одноразовых кодов авторизации.
- **Маркеры аномалии:** В логах резко возрастает число операций со статусом `Мошенническая`, где методом подтверждения является `SMS-код`, а тип операции — `Перевод физлицу` с аномально крупным чеком.
- **Действия аналитика:** Локализация уязвимости через кросс-матрицу распределения фрода, формирование рапорта для СБ с директивой о временном принудительном переводе данных клиентов на биометрическую верификацию.

### Сценарий 4: Искусственное дробление сумм (Обход лимитов комплаенса)
- **Суть инцидента:** Попытка обойти внутренние лимиты обязательного контроля банка путем проведения серии мелких транзакций вместо одной крупной.
- **Маркеры аномалии:** Нарушение закона криминалистического распределения первой цифры на графике Бенфорда (аномальный всплеск частоты на цифрах 4 или 5 при резком падении единиц).
- **Действия аналитика:** С помощью вертикального ползунка сумм изоляция бакета подозрительного диапазона (например, `$400 – $499`), выявление целевого шлюза эквайринга и экспорт реестра для передачи в СБ.

### Сценарий 5: Таргетированная атака на уязвимый протокол 3D-Secure v1
- **Вектор атаки:** Использование хакерами устаревших расчетных шлюзов мерчантов, поддерживающих небезопасную первую версию протокола 3D-Secure.
- **Маркеры аномалии:** На графике «Уязвимости двухфакторной верификации» столбец `3D-Secure v1` окрашивается преимущественно в красный цвет (высокая плотность фрода).
- **Действия аналитика:** Выбор шлюза в выпадающем списке, автоматический сбор всех логов по уязвимому протоколу и отправка официального уведомления в ФинЦЕРТ ЦБ РФ для отключения скомпрометированного эквайера.

### Сценарий 6: Верификация высокорисковых транзакций без подтверждения (Zero-Auth Fraud)
- **Вектор атаки:** Попытка проведения мошеннических списаний со скомпрометированных карт через зарубежные сервисы, не требующие подтверждения операции (No-3DS транзакции).
- **Маркеры аномалии:** Категория перевода — `Онлайн-покупка`, защита сессии — `Без подтверждения`, а уровень аппаратного риска устройства превышает `0.75`.
- **Действия аналитика:** Использование живых фильтров для мгновенного сужения таблицы до No-3DS операций, выделение аномального бакета рисков и выгрузка специального сообщения для Росфинмониторинга по 115-ФЗ.

### Сценарий 7: Мониторинг аномальной активности в шлюзе СБП (Система быстрых платежей)
- **Суть инцидента:** Мошенники используют шлюз СБП для мгновенного невозвратного вывода похищенных средств со счетов физических лиц.
- **Маркеры аномалии:** Канал эквайринга — `СБП`, тип операции — `Перевод физлицу`, тренд количества инцидентов показывает резкий вертикальный взлет.
- **Действия аналитика:** Выделение всех фрод-сессий по каналу СБП через кнопки массового выбора строк, выгрузка изолированного CSV-файла и передача данных в межбанковскую систему обмена киберугрозами для блокировки карт-дропов.

### Сценарий 8: Компрометация цифровых отпечатков устройств (Device Spoofing)
- **Вектор атаки:** Хакеры используют специализированный софт для подмены параметров железа и цифрового отпечатка браузера (Anti-detect browsers) с целью симуляции «чистого» устройства клиента.
- **Маркеры аномалии:** Показатель `Индекс аномалии лога` (в расширенном датафрейме) находится на пиковых значениях при низком или среднем базовом риске устройства.
- **Действия аналитика:** Раскрытие выплывающей панели, настройка диапазона индекса аномалий, точечный отбор подозрительных сессий чекбоксами и экспорт данных в Excel для детального технического разбора ИБ-инженерами.

### Сценарий 9: Выявление скрытых транзакций по отмыванию доходов (AML-мониторинг)
- **Суть инцидента:** Использование банковской инфраструктуры организованными группами для транзита и легализации незаконных доходов через криптовалютные обменники.
- **Маркеры аномалии:** Огромный оборот по каналу `Криптовалюта`, проходящий через категорию `Перевод физлицу` со стабильно высоким чеком.
- **Действия аналитика:** Использование правила синхронизации — выставить «Криптовалюту» в сайдбаре, сузить диапазон сумм в таблице до крупных переводов, сформировать печатную форму отчета 115-ФЗ для последующей передачи дела в Росфинмониторинг.

### Сценарий 10: Стресс-тестирование точности SIEM-правил на исторических логах
- **Суть процесса:** Регулярная проверка эффективности работы антифрод-системы, оценка уровня ложноположительных срабатываний при обновлении правил фильтрации.
- **Маркеры аномалии:** Изменение метрики `Точность правила` на панели бизнес-моделирования при динамическом движении слайдера порога жесткой блокировки.
- **Действия аналитика:** Пошаговое изменение порога блокировки от минимального к максимальному, фиксация изменений графиков распределения, печать сводной матрицы финансовых потерь и формирование докладной записки для руководства ИБ о переходе на новые весовые коэффициенты скоринга.
